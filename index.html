myGame of Life

This is my crack at a JavaScript-backed implementation of John Conway's ageless Game of Life.

    The rules--as much for my benefit as yours--are as follows:
    For every generation
  • Any live cell with fewer than two live neighbors dies
  • Any live cell with more than three live neighbors dies
  • Any live cell with two or three live neighbors survives
  • Any dead cell with exactly three live neighbors becomes alive
 
    The modes of interaction with this board are as follows:
  • Left-click any empty cell to 'raise' it--to make it alive
  • Left-click any live cell to kill it
  • Left-click and drag to either batch raise or batch kill cells
  • Right-click and drag anywhere on the board to move the view frustum

My own edification, is the short answer. I've had, for a long time, a cursory understanding of HTML, CSS, and JavaScript, but I've never been deeply acquainted with them, nor had I before worked with HTML5 and CSS3 and the 'best practices' that come along with them. There's a trend in the industry that seems to be favoring using these technologies for new, adaptable, and clean user-interfaces, and I wanted to know why.

I must say, it is very easy to make broad changes in the structure of this interface with surprisingly few lines of code. It remains my (entirely correct) opinion that HTML is outmoded, the focus of CSS is backwards (too much specificity in all the wrong places), and JavaScript was designed by a committee in under a month. (okay, that last one's not an opinion; it's what actually happened. Pretty telling, though, yeah?) As such, the secondary focus of this experiment was to familiarize myself with CoffeeScript and Sass, so I might know what coding for the web in a sane environment could be like.

I did my best to follow web-coding "best practices." (I put "best practices" in quotations, not because I don't want my structure, style and code to be among the best out there, but because it's my experiences that these best practices are motivated by decades-old workaround, and petty squabbles.)

I did my best to keep the markup of this page sane, but it doesn't much use `article`, `section`, or other such specific tags. This kind of UI doesn't lend itself to index searches or Readability to begin with, so I figured a few extra `div` tags wouldn't hurt much.

I wanted to do as much of the coloring, positioning, and user-interaction of textural elements in CSS as I could; I'm somewhat proud of the results. The slider is (unfortunately) JQuery because of some `mouseup` resolution oddities, as is the animation of the fading text. The rest is pure CSS. The hovering, selecting and transitioning of the buttons and tabs were put together in Sass (scss, specifically, because I love me some supersets) and compiled to pure CSS.

The Canvas was heavily inspired by the excellent PathFinding.js library from Xueqiao Xu. It uses a Raphael.js canvas for rendering simple vector graphics and JQuery for properly capturing user input. The rest is a rather simple MCV program hashed together in CoffeeScript.

Because I'm incredibly lazy, I had to put together a Watchr script to automate compiling both the CoffeeScript down to JavaScript, and the Sass down to CSS.
